# Промежуточная диагностика

## Выявленные причины

1. После удаления `.php` проект остался жёстко привязан к корню домена через абсолютные пути вида `/css/...`, `/api/...`, `/admin/...`, `/item/...`.
2. В `.htaccess` была указана `RewriteBase /`, что ломает работу сайта при установке в подпапку.
3. Редиректы админки после входа и выхода были жёстко заданы как `/admin/`, из-за чего при запуске не из корня происходил уход на неверный адрес.
4. Бесконечная загрузка публичной части, вероятнее всего, была связана с тем, что фронтенд не мог достучаться до API по адресу `/api/...` вне корня домена.
5. Для блока «Хиты продаж» использовался флаг `popular`; добавлена серверная совместимость с альтернативными полями `hit`, `is_hit`, `featured`.

## Уже внесённые исправления

- Добавлены `getBasePath()`, `appUrl()`, `rewriteAppUrls()` и `startAppOutputBuffer()` в `includes/functions.php`.
- Включена глобальная корректировка путей в `index.php` и `admin/index.php`.
- Исправлены редиректы админки на динамический путь через `appUrl('/admin/')`.
- В `main.js` добавлен helper `appUrl()` и исправлены ключевые запросы к API, ссылки на оплату и загрузка иконок.
- В `.htaccess` убрана жёсткая привязка `RewriteBase /`.

## Что осталось проверить

- PHP lint по всему проекту.
- Поведение главной, карточки товара, категории, оплаты и админки после правок.
- Возможные дополнительные абсолютные пути в inline-скриптах и шаблонах.
- Полная проверка функциональности и финальная упаковка проекта.

## Дополнительные выводы по тестированию

Проверка страницы товара показала, что сам шаблон товара работает корректно: по query-маршруту `index.php?page=item&slug=gmail-fresh-br-998` страница открывается, карточка рендерится, количество меняется, кнопки «Купить сейчас» и «В корзину» присутствуют. При этом проверка красивого адреса `/item/.../` на встроенном PHP-сервере даёт некорректную картину, потому что локальный сервер без отдельного router-скрипта не воспроизводит правила `.htaccess` полностью. Следовательно, для pretty URL нужен либо Apache/Nginx, либо router-обвязка для встроенного сервера.

Это означает, что визуальная проблема шаблона товара не подтверждается, а основной риск остаётся в серверном rewrite-слое и жёстких абсолютных путях. Следующим шагом нужно проверить публичный API и сценарий оплаты, а затем упаковать все исправления.

Проверка страницы оформления заказа через локальный query-маршрут показала, что сама форма загружается корректно: товар подтягивается, сумма считается, email-поля и выбор метода оплаты отображаются без ошибок. Однако при тестовом переходе к оплате через YooMoney открывается страница `quickpay/error?reason=referer` с сообщением, что переводы недоступны на этом сайте. Это уже не ошибка CSS или фронтенд-маршрутизации, а отдельный дефект в текущей схеме интеграции оплаты или в разрешённых доменах формы YooMoney.

Нужно отдельно проверить код генерации платёжной ссылки и параметры merchant/receiver/successURL, чтобы понять, является ли проблема локальным ограничением домена 127.0.0.1 или дефектом, который затронет и боевой домен.

Повторная браузерная проверка админки после правок подтвердила, что после авторизации интерфейс открывается корректно: виден дашборд, список разделов, статистика, таблица последних товаров и модальные формы. Белый экран после логина больше не воспроизводится. Дополнительная проверка консоли страницы не показала JavaScript-ошибок, то есть критический сбой клиентского кода в текущем состоянии устранён.

Проверка главной страницы в браузере показала, что стили снова загружаются корректно: видны оформленные хедер, hero-блок, кнопки, карточки и подвал, признаков «голого HTML» и бесконечного прелоадера нет. Поиск по странице подтвердил наличие секции «Хиты продаж — Аккаунты». Это согласуется с серверной логикой getPopularProducts(), которая теперь выбирает на главную только активные товары с установленным флагом popular/hit/is_hit/featured.

Проверка оформления заказа через локальный API в демо-режиме завершилась успешно. Корректный формат payload для корзины — `items: [{ product_id, qty }]`. Заказ был создан, получил статус `demo-paid`, а товар был автоматически выдан в рамках демо-сценария. Это подтверждает, что ключевая цепочка «товар → корзина → заказ → выдача» после правок работает.

Переход на реальную форму YooMoney из локального стенда по-прежнему может завершаться внешней ошибкой `referer`, что связано с ограничениями тестового локального запуска и политикой внешнего платёжного шлюза, а не с падением фронтенда или внутреннего API проекта.
