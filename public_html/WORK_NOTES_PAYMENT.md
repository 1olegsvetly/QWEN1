# Рабочие заметки по криптооплате

## Что уже подтверждено

1. Проект работает на PHP + HTML/CSS/JS и хранит данные в JSON-файлах, без базы данных.
2. Текущая ветка `crypto` в `api/index.php` является заглушкой (`getCryptoPlaceholderData()`), её нужно заменить на рабочую логику.
3. Макет из `Untitled 1.png` содержит два целевых экрана:
   - экран выбора токена и суммы;
   - экран оплаты с QR, адресом кошелька, суммой и статусом `Ожидает оплату`.
4. Из QR-кодов извлечены реальные адреса кошельков, они сохранены в `CRYPTO_QR_MAPPING.md`.
5. Большой QR на экране 2 в макете использует EVM-адрес `0xF03f3009663Aa1eA87266d97f9FE6C8eFdDE39aa`.

## Источник курсов

Для локального кэша курсов подходит endpoint CoinGecko Simple Price:
- документация: https://docs.coingecko.com/reference/simple-price
- позволяет получать цены нескольких монет в `USD`;
- поддерживает `include_last_updated_at=true`;
- значит можно обновлять локальный JSON-кэш раз в 12 часов и затем считать сумму оплаты локально.

## Предварительная техническая схема

- хранить локальный файл с курсами в `data/crypto_rates.json`;
- при создании криптозаказа обновлять кэш только если он старше 12 часов;
- для каждой валюты хранить:
  - код токена;
  - адрес кошелька;
  - тип сети/эксплорера;
  - id монеты для курсов;
  - число подтверждений;
- после выбора токена создавать заказ со снимком курса и точной суммой токена;
- на экране оплаты показывать QR, адрес, точную сумму, таймер/срок действия и статус ожидания;
- статус подтверждать через проверку блокчейн-транзакций по адресу, сумме и окну времени, затем выполнять выдачу товара.

## Дополнение по конвертации RUB → USD

Для расчёта суммы криптооплаты из рублёвой цены товара нужен локальный курс доллара к рублю. Подтверждён удобный JSON-источник `https://www.cbr-xml-daily.ru/daily_json.js`.

На главной странице сервиса опубликован JSON с `base = RUB`, где поле `rates.USD` означает количество долларов за 1 рубль. Следовательно:
- `usd_per_rub = rates.USD`;
- `rub_per_usd = 1 / rates.USD`.

Это позволяет хранить в одном локальном кэше:
- курс USD/RUB;
- курсы токенов к USD;
- итоговые суммы токенов для оплаты считать локально без постоянных внешних запросов.

## Визуальная проверка экрана 1 — 2026-04-11 19:25

Открыт локальный маршрут `/?page=oplata&item=instagram&qty=1`.

Ключевые наблюдения:
- новый экран выбора токена успешно подключён и рендерится как отдельная страница `Криптооплата`;
- отображаются поля email, подтверждение email, изменение количества, сетка токенов, блок суммы и правая колонка заказа;
- локальный кэш курсов подставляется корректно, но для тестового URL товар `instagram` не найден в данных проекта, поэтому правая колонка показывает пустую корзину и сумма оплаты равна `0`;
- в настройках проекта криптооплата сейчас выключена, поэтому на странице выводится предупреждение о необходимости включить метод оплаты;
- визуальный каркас первого экрана уже готов и подходит как база для следующего этапа: экрана 2 с QR-кодом, статусом ожидания оплаты и автопроверкой транзакции.


Повторная проверка на реальном товаре `aged-93-798` с количеством `2` подтвердила, что первый экран уже считает сумму корректно: `1 268 ₽`, `$16.47` и эквиваленты по всем токенам. Правая колонка показывает конкретный товар, количество и итоговую сумму, а сетка токенов заполняется из локального кэша курсов. Это означает, что этап экрана 1 можно считать готовым как основу для следующего шага — экрана 2 с QR-кодом, ожиданием оплаты и проверкой транзакции.

## Подтверждённые API для верификации криптооплаты

- **TRONSCAN**: документация `https://docs.tronscan.org/en/api/transactions-and-transfers` подтверждает публичные endpoint'ы `https://apilist.tronscanapi.com/api/transfer` для входящих **TRX/TRC10** переводов по адресу и `https://apilist.tronscanapi.com/api/token_trc20/transfers` / `token_trc20/transfers-with-status` для **TRC20** переводов. Это подходит для проверки `TRX` и `USDT_TRC20`.
- **Etherscan API v2**: документация `https://docs.etherscan.io/api-reference/endpoint/tokentx` подтверждает endpoint `GET /v2/api` с параметрами `module=account`, `action=tokentx`, `contractaddress`, `address`, `startblock`, `endblock`, `page`, `offset`, `sort` для проверки входящих **ERC-20** переводов на адрес. Для нативных переводов доступен соседний endpoint `Get Normal Transactions By Address`.
- Для сетей **BTC/BCH/LTC/DOGE/DASH** можно использовать адресные endpoint'ы Blockchair, но страницу docs пришлось открывать через защиту, поэтому при реализации лучше опираться на прямые HTTP-проверки реальных JSON-ответов.
- Для **TON** и **Solana** нужно отдельно подтвердить рабочие JSON endpoint'ы через прямые запросы или документацию перед серверной интеграцией.

Проверка [Tokenview](https://tokenview.io/) не дала практического результата для текущей интеграции: сайт уходит в Cloudflare Security Verification, поэтому использовать его как основной источник серверной верификации входящих платежей в автоматическом режиме нецелесообразно.

Прямая проверка JSON endpoint Blockchair для BTC-адреса показала ответ с кодом **430** и сообщением о временной блокировке IP из-за превышения лимитов. Значит, Blockchair нельзя брать как основной механизм автоматической верификации для боевой логики этого проекта без собственного ключа и контроля лимитов.

Документация [Blockchain.com API](https://www.blockchain.com/explorer/api/blockchain_api) подтверждает пригодный для **BTC** endpoint `https://blockchain.info/rawaddr/{address}`. Он возвращает `txs`, `total_received`, `final_balance` и позволяет проверять входящие переводы по адресу без собственного узла.

Проверка [SoChain API](https://chain.so/api/) показала, что хотя у сервиса есть endpoint получения транзакций по адресу (`/api/v3/transactions/{network}/{address}/{page}`), для доступа нужен отдельный **API Key**. Поэтому SoChain не подходит как полностью самодостаточный публичный источник для этой сборки без дополнительных секретов.
